# LinMinheng · 使用手册

这是一个静态个人博客。内容保存在仓库文件里,由 Astro 构建成静态页面,再部署到 Cloudflare Pages。

这套方式的公开网站是静态页面,没有传统数据库后台。内容通过仓库文件管理,也可以用 `/admin/` 里的 Git CMS 编辑,保存时仍然提交到 GitHub。文章和个人资料都能被 Git 记录、回滚和备份,长期维护更稳。

### 最省事的网页管理方式

项目里预留了一个 Git CMS:

```txt
/admin/
```

它不是数据库后台,而是一个网页表单编辑器。它编辑 Markdown/JSON 文件,保存时提交到 GitHub,再由自动部署更新网站。

后台入口有一层管理员密码,当前临时密码是:

```txt
lmh20030805
```

这层密码用于挡住普通访客。更严格的线上保护建议在 Cloudflare Zero Trust 里给 `/admin/*` 配 Cloudflare Access。真正能不能保存内容,仍然取决于 GitHub 登录和仓库权限。

第一次使用需要在 GitHub 上确认登录和仓库授权。这个方式适合在外面用浏览器管理博文、项目、站点和个人资料。

如果点击 `Sign in with GitHub` 出现 OAuth 配置错误,需要先创建 GitHub OAuth App:

```txt
Homepage URL: https://blog.loven7.com
Authorization callback URL: https://blog.loven7.com/admin/
```

然后在 Cloudflare Pages 的环境变量里添加:

```txt
GITHUB_CLIENT_ID=你的 GitHub OAuth App Client ID
```

可选添加:

```txt
GITHUB_AUTH_SCOPE=public_repo read:user
```

这个项目现在使用 GitHub Device Flow,不需要 `Client Secret`。登录时会显示一个 GitHub 授权码,按页面提示打开 GitHub 确认即可。如果没有配置 `GITHUB_CLIENT_ID`,项目会使用代码里的公开兜底 Client ID,但长期维护建议改成自己的 OAuth App。

如果登录页提示 `Device Flow` 没启用,进入 GitHub OAuth App 设置,勾选:

```txt
Enable Device Flow
```

保存后重新部署一次。配置完成后,后台 GitHub 登录会走 `https://blog.loven7.com/api/auth`,不会再跳到 Netlify。

后台编辑器使用项目内固定版本的 Sveltia CMS:

```txt
public/vendor/sveltia-cms-0.167.3.js
```

升级 CMS 时,替换这个文件,同步修改 `src/pages/admin.astro` 里的文件名,然后运行 `npm run qa`。不要直接改回 CDN 地址,否则外部网络异常时后台会更容易打不开。

### 在外面快速新建文章

如果人在外面,没有本地环境,不用复制模板。

打开 GitHub 仓库后:

1. 点 `Actions`
2. 选择 `New blog post`
3. 点 `Run workflow`
4. 填标题、标签、摘要
5. 提交后 GitHub 会自动创建一篇 Markdown 草稿

GitHub Action 创建的文章默认会直接公开发布。只有手动选择保存为草稿时,它才会写成:

```txt
draft: true
```

草稿不会显示在首页、博客列表、RSS 和站点地图。要公开时把它改回 `draft: false`,再提交一次,网站就会自动发布。

如果是在 `/admin/` 后台新建文章,现在默认会直接公开发布。只有手动打开“保存为草稿（不公开）”时,文章才会隐藏。

文章放在:

```txt
src/content/blog/
```

一篇文章一个 `.md` 文件。建议用英文或日期命名,例如:

```txt
2026-06-19-blog-redesign.md
```

模板:

```markdown
---
title: "文章标题"
date: 2026-06-19
updated: 2026-06-19
tags: ["博客", "独立开发"]
excerpt: "文章列表里显示的摘要。"
draft: false
foot: "可选的结尾小字"
---

正文从这里开始。
```

常用字段:

- `title`:文章标题
- `date`:发布日期
- `updated`:最后更新日期,可选
- `tags`:标签
- `excerpt`:摘要
- `slug`:文章短链接,可选。建议使用英文小写、数字和连字符,例如 `first-post`;不填时网站会自动根据文件名生成
- `draft`:是否草稿。`true` 不会出现在首页、博客列表、RSS 和站点地图
- `foot`:文章结尾的小字,可选
- `cover`:封面图路径,可选

### 在文章里插入图片

在后台的“正文”编辑区里,直接粘贴一条图片链接即可,不要点工具栏里的图片上传弹窗。例如:

```txt
https://img.loven7.com/file/example.JPG
```

编辑器会自动转换成 Markdown 图片,右侧预览会显示图片。前台也支持单独一行图片链接自动渲染成图片。`img.loven7.com/file/...` 这种图床链接即使没有 `.jpg` 后缀也可以识别。

常见图片后缀会自动识别,包括 `jpg`、`jpeg`、`png`、`webp`、`gif`、`avif`、`svg`、`bmp`、`ico`、`jfif`、`tif`、`tiff`、`heic`、`heif`。其中 `heic`、`heif` 是否能显示,还取决于访客浏览器本身是否支持。

如果需要手动写,格式是:

```markdown
![图片说明](https://img.loven7.com/file/example.JPG)
```

需要控制宽度或比例时,可以用 HTML:

```html
<figure class="image-medium image-center ratio-4x3">
  <img src="https://img.loven7.com/file/example.JPG" alt="图片说明" />
  <figcaption>图片说明。</figcaption>
</figure>
```

可用类名:

- `image-medium`:中等宽度
- `image-narrow`:窄宽度
- `image-center`:居中
- `ratio-16x9` / `ratio-4x3` / `ratio-1x1`:固定图片比例

```bash

```

自动部署需要 GitHub 仓库里配置 `CLOUDFLARE_API_TOKEN` 这个 secret。GitHub Actions 会先执行 `npm run qa`,通过类型检查、内容校验、构建、安全审计和浏览器检查后才部署。

## 访客统计

网站有一个低调的访客统计,显示的是全站唯一访客数。它按 IP 去重:同一个 IP 多次打开网站,只会计入 1 个访客。

为了隐私,接口不会保存原始 IP。它会把 IP 加盐后生成哈希,数据库里只保存哈希值、首次访问时间、最后访问时间、访问次数和 Cloudflare 提供的聚合地点字段。

地点统计只保存聚合维度:

```txt
国家 / 地区 / 城市 / Cloudflare 机房 / 时区
```

不会保存 IP 明文,也不会在公开页面暴露访客明细。

推荐绑定 Cloudflare KV,不占用 D1 数据库:

```txt
KV 命名空间建议名: linminheng-blog-visitors
Pages 绑定名: VISITOR_KV
```

KV 适合这种小型访客计数,维护成本低。它不是事务型数据库,极端并发时计数可能有轻微误差,但个人博客场景通常足够。

如果想要更严格的事务统计,也可以绑定 D1 数据库作为兜底:

```txt
数据库建议名: linminheng-blog-visitors
Pages 绑定名: VISITOR_DB
```

初始化数据库时执行:

```bash
npx wrangler d1 migrations apply linminheng-blog-visitors --remote
```

可选:在 Cloudflare Pages 的环境变量里添加 `VISITOR_SALT`,用于生成更私有的访客哈希。设置后不要频繁更换,否则旧访客会被当作新访客重新统计。

如果要读取地点统计报告,需要额外设置:

```txt
VISITOR_REPORT_TOKEN=一个足够长的随机密钥
```

然后用:

```bash
curl -H "Authorization: Bearer 你的密钥" https://blog.loven7.com/api/visitor-report
```

没有配置 `VISITOR_REPORT_TOKEN` 时,报告接口不会启用。地点统计从功能上线后的新访问开始累积,历史访客不能反推出 IP 或地点。

如果同时绑定了 `VISITOR_KV` 和 `VISITOR_DB`,接口会优先使用 KV。如果两个都没绑定,页面不会报错,访客数字会自动隐藏。

## 文件结构

```txt
site/
├─ src/
│  ├─ content/blog/      # 博文 Markdown
│  ├─ data/              # 个人信息、项目、站点、历程
│  ├─ layouts/Base.astro # 页面外壳、主题切换和交互脚本
│  ├─ pages/index.astro  # 首页
│  └─ styles/global.css  # 全站样式
└─ public/assets/        # 图片和 logo
```

## 出问题时

- 构建失败:优先检查 Markdown 头部字段、日期格式、数组逗号
- 页面没更新:先确认本地 `npm run qa` 成功,再重新发布
- 想回到旧版本:用 Git 历史回滚
